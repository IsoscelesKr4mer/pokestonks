# Pokestonks DB-First Search Design

**Date:** 2026-04-26
**Status:** Draft, pending user review
**Parent spec:** `docs/superpowers/specs/2026-04-25-pokestonks-design.md`
**Predecessor:** `docs/superpowers/plans/2026-04-25-pokestonks-plan-2-catalog-search-images.md` (Plan 2 main pass — shipped)
**Successor:** writing-plans skill consumes this design and produces the executable plan.

## 1. Goal

Subsequent searches of a query that's been run before should return instantly from `catalog_items` instead of re-fetching from Pokémon TCG API + TCGCSV. The user gets an explicit Refresh button to force an upstream fetch when they want fresher prices. Closes the "lazy hybrid" promise from the original Plan 2 design.

This is the Plan 2 capstone — the search service started lazy-importing rows on Day 1, but never read from those imports on subsequent searches. After this iteration, Plan 2 is truly done.

## 2. Decisions baked in (from brainstorming)

| Decision | Choice | Why |
|---|---|---|
| Freshness policy | Always serve local; explicit Refresh button | Keeps the mental model simple. User controls when to pay the upstream cost. |
| Schema location for cached price | Denormalize: `last_market_cents` + `last_market_at` columns on `catalog_items` | Single-table reads, no joins. `market_prices` stays clean for Plan 6's price chart history. |
| Local query mechanics | Postgres `ILIKE` per-token, OR-across-fields, AND-across-tokens | Mirrors the existing upstream tokenizer. ~10K rows max for the foreseeable future, so even a sequential scan is sub-100ms. |
| Fallback to upstream | Only on zero local matches | Refresh button covers the partial-results edge case. Avoids extra tracking complexity. |

## 3. Architecture

### 3.1 New surfaces

```
lib/services/
  searchLocal.ts                # NEW — read catalog_items via ILIKE, returns DTOs
  search.ts                     # MODIFIED — searchAll calls local first, falls back to upstream
  
lib/db/upserts/
  catalogItems.ts               # MODIFIED — bulkUpsertCards/upsertSealed write last_market_cents + last_market_at

app/api/
  search/route.ts               # MODIFIED — local-first dispatch in GET
  search/refresh/route.ts       # NEW — POST that always runs upstream + bulk-upsert

components/catalog/
  SearchBox.tsx                 # MODIFIED — Refresh button, "as of" caption
  RefreshButton.tsx             # NEW — small dedicated component (icon + spinner)

drizzle/
  XXXX_*.sql                    # NEW — generated migration for the two columns
```

### 3.2 Data flow

```
GET /api/search?q=ascended+heroes&kind=all&sortBy=price-desc&limit=500
  ↓
search.ts:searchAll
  ├─ tokenizeQuery(q)
  ├─ searchLocalCatalog(tokens, kind, limit, sortBy)
  ├─ if (results.length > 0): return them                  ← FAST PATH (~200ms)
  └─ else: existing upstream fetch + bulk-upsert + DTO     ← SLOW PATH (~5s)
                                  ↑
                                  also writes last_market_cents
                                  + last_market_at on every row

POST /api/search/refresh?q=ascended+heroes&kind=all&sortBy=price-desc&limit=500
  ↓
  Always runs the upstream slow path. Headers: Cache-Control: no-store.
```

## 4. Schema migration

```sql
ALTER TABLE catalog_items
  ADD COLUMN last_market_cents INTEGER,
  ADD COLUMN last_market_at TIMESTAMPTZ;
```

Both columns nullable. Existing rows keep `NULL` for both until their next upstream-fetch reaches them. No backfill — those rows already display `—` for price under current behavior.

No new index in this migration. If sorting by `last_market_cents DESC` becomes slow at scale we can add `CREATE INDEX ON catalog_items (last_market_cents DESC NULLS LAST);` later.

Drizzle schema update mirrors the SQL (`lastMarketCents: integer(...)`, `lastMarketAt: timestamp(... { withTimezone: true })`). Migration generated by `npm run db:generate` and applied via `npm run db:migrate` (the `db:push` path needs a TTY and breaks in non-interactive contexts).

`CardUpsertInput` and `SealedUpsertInput` types in `lib/db/upserts/catalogItems.ts` gain a `lastMarketCents: number | null` field. The bulk upsert writes `last_market_at = NOW()` server-side via `sql\`NOW()\`` on every conflict-update so callers don't need to pass timestamps.

## 5. Local search query (`lib/services/searchLocal.ts`)

Single exported function:

```ts
export async function searchLocalCatalog(
  tokens: Tokens,
  kind: SearchKind,
  limit: number,
  sortBy: SortBy
): Promise<{ sealed: SealedResultDto[]; cards: CardResultDto[]; warnings: Warning[] }>;
```

### 5.1 Predicate construction

For each token class, an AND-clause:

| Token class | SQL |
|---|---|
| `text` token (each, OR-across-fields, AND-across-tokens) | `(name ILIKE '%X%' OR set_name ILIKE '%X%')` |
| `cardNumberFull` (e.g. `199/091`) | `(card_number = '199/091' OR card_number LIKE '199/%')` |
| `cardNumberPartial` (e.g. `199`) | `(card_number = '199' OR card_number LIKE '199/%')` |
| `setCode` (e.g. `sv3pt5`) | `set_code = 'sv3pt5'` |
| `kind = 'sealed'` filter | `kind = 'sealed'` |
| `kind = 'card'` filter | `kind = 'card'` |
| `kind = 'all'` | (no kind predicate) |

Built via Drizzle's `and()` / `or()` so we get parameter binding rather than string concat.

### 5.2 Result mapping

Each row maps to a DTO using the same shape `searchAll` produces upstream:
- `imageUrl` resolved through `getImageUrl({ imageStoragePath, imageUrl })` (cached storage URL when present).
- `marketCents` = `row.last_market_cents`.
- DTO `type` set from `row.kind`.

### 5.3 Sorting

After fetching, sort in JS via the existing `applySort` helper from `search.ts`. The helper is currently file-local and gets exported as part of this work so `searchLocal.ts` can import it. Catalog scope is small; pulling all matching rows and sorting in memory is cheap.

### 5.4 Empty handling

Returns `{ sealed: [], cards: [], warnings: [] }` when nothing matches. The `searchAll` caller sees the empty union and falls through to the upstream path.

### 5.5 No abbreviation expansion in local

Upstream's `searchSealed` expands `etb` → "elite trainer box" before matching. The local layer doesn't; it expects whole words. Once a sealed product has been imported via upstream, plain ILIKE on the real product name works for any future search. The expansion still happens on the upstream-fallback path so first-ever searches behave as today.

## 6. Refresh flow

### 6.1 Endpoint

`POST /api/search/refresh` with the same query params as `GET /api/search`:

```
POST /api/search/refresh?q=ascended+heroes&kind=all&sortBy=price-desc&limit=500
```

Validated by the same Zod schema as the GET route.

### 6.2 Behavior

1. Auth-gated by middleware (existing).
2. Skips `searchLocalCatalog` entirely.
3. Runs the existing upstream `searchAll` slow path (Pokémon TCG API + TCGCSV merge + bulk upsert).
4. Bulk upsert writes `last_market_cents` and `last_market_at = NOW()` along with everything else.
5. Returns the same `SearchResponse` body shape with `Cache-Control: no-store`.

### 6.3 Concurrency

Two refresh calls for the same query in flight at once (e.g., double-click) both run. The upsert is idempotent on the partial unique index, so the second call overwrites with identical data. Acceptable.

## 7. /api/search GET wiring

The route keeps the same query schema. Inside the handler:

```ts
const tokens = tokenizeQuery(q);
const local = await searchLocalCatalog(tokens, kind, limit, sortBy);
if (local.sealed.length + local.cards.length > 0) {
  return NextResponse.json({
    query, kind, sortBy,
    results: applySort([...local.sealed, ...local.cards], sortBy).slice(0, limit),
    warnings: local.warnings,
    source: 'local',
  });
}
const upstream = await searchAll(q, kind, limit, sortBy);
return NextResponse.json({ ...upstream, source: 'upstream' });
```

`source` field on the response is for diagnostics — frontend can log or display it; doesn't drive UI logic.

`Cache-Control` stays `private, max-age=300, stale-while-revalidate=1800` for the local path. The browser cache on top of local DB is fine; both layers are cheap to invalidate via Refresh.

## 8. Frontend changes

### 8.1 Refresh button

New `components/catalog/RefreshButton.tsx`:
- Icon-only button (refresh / circular-arrow icon) next to the sort dropdown.
- Disabled when no query is active.
- On click: fires `POST /api/search/refresh?...` with the current query/kind/sortBy. While in flight, swaps the icon for a spinner.
- On success: invalidates the TanStack Query cache for the matching `['search', q, kind, sortBy]` key (and effectively forces a re-fetch via the GET route, which now reads the freshly-updated rows).

### 8.2 "As of" caption

Below the result count line, a small muted caption reads:
- `Updated just now` (if the oldest `last_market_at` among visible results is < 5 min)
- `Updated 2 hours ago` (relative time formatting)
- `Last updated unknown` (if any visible row has `last_market_at = NULL`)

Computed from the `last_market_at` field carried on each result DTO. The existing DTOs need `lastMarketAt` added so the frontend can render the caption — that's a small DTO field addition.

### 9.2 Manual acceptance

- [ ] First-ever search of "destined rivals" (or any never-imported set): ~5s, populates DB, returns results.
- [ ] Second search of same query, fresh tab: under ~500ms, served from local.
- [ ] Click Refresh: spinner shows, ~5s wait, fresh prices arrive, "Updated just now" appears.
- [ ] Search "ascended heroes pikachu" after AH was imported: instant.
- [ ] Sort by price after refresh: highest-priced cards still on top.
- [ ] DB check: every searched query's rows have `last_market_at` set to a recent timestamp.

## 10. Coding conventions reminders

- Money as integer cents end-to-end. `last_market_cents INTEGER`. `formatCents` only at render.
- Dates as ISO `YYYY-MM-DD` strings or Postgres `DATE` for purchase/sale dates. `last_market_at` is `TIMESTAMPTZ` (it's a real wall-clock event time).
- No em-dashes in user-facing copy.
- Drizzle for service-role / public-read contexts. `searchLocalCatalog` reads `catalog_items` which is public-read under RLS, so Drizzle is fine.
- No raw `fetch` in components; the Refresh button uses TanStack Query's `useMutation`.

## 11. Done means

1. Migration applied: `last_market_cents` + `last_market_at` columns live on `catalog_items` in production.
2. `searchLocalCatalog` lands rows from `catalog_items` for any query that's been run before.
3. `/api/search` GET serves local first, falls back to upstream on zero matches.
4. `/api/search/refresh` POST always hits upstream and updates rows.
5. SearchBox has a Refresh button and a relative-time "Updated" caption.
6. All tests pass.
7. Manual acceptance: first-ever search slow, every subsequent search instant, Refresh forces fresh.

When all of the above are true, **Plan 2 is truly complete** and Plan 3 (Purchases) becomes the next focus.

## 12. Out of scope (Plan 6+)

- Daily price refresh cron (background batch refresh of all imported rows).
- Manual catalog entry UI.
- Trigram / fuzzy matching for typo-tolerant local search.
- Pre-fetching popular sets on app deploy.
- Eviction of stale catalog rows.
